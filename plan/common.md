# phase-fiction-skill 通用规划约束

本文件是 phase-fiction-skill 全部 phase 的长期稳定约束来源。任何单步执行都必须把本文件作为完整上下文的一部分，而不是只看局部任务。

## 结论

本项目的目标不是给“代码迁移/工程重构”继续套一个新名字，而是把现有仓库彻底改造成一份面向长篇小说创作、续写与修订的长任务 Skill：它必须帮助 AI 在长时间跨度里稳定推进题材定位、叙事引擎、人物弧光、章节结构、场景调度、悬念铺设、语言润色与终稿收束。

本规划将以下要求视为硬约束，而不是“后续优化项”：

- 顶层品牌统一改为 phase-fiction-skill，不保留 phase-contract-workflow 作为主名称。
- 核心方法论必须锚定小说创作，不得停留在“通用写作建议”或“营销文案生成”。
- 小说方法论必须覆盖 premise、人物欲望与缺口、冲突链、赌注升级、情节引擎、场景设计、悬念与回报、节奏控制、叙事视角、主题/母题、修订闭环等关键维度。
- 所有 phase 合同与模板必须强调可操作、可判断、可续跑，避免含糊的“写得更好”“更精彩”之类主观空话。
- 顶层 README 保持概括性，细节优先沉淀到 references/* 并通过链接引用。
- 中英文顶层文档需要同步改造，避免仅英文或仅中文完成而导致发现入口失真。
- 保留磁盘外部化、manifest/state/handoff、三文件上下文律与 planctl 调度这套基础机制；除非确有必要，不改其底层工作方式。
- 不引入新的运行时依赖；继续保持 Ruby 单文件脚本与现有仓库结构可用。
- 三份 agent 指令文件必须字节一致，禁止 agent-specific 漂移。
- 验证至少包含 planctl 的 status / doctor / advance 检查，以及与改动切面相关的测试或语义校验。

## 产品/项目目标

交付一份可以被 Copilot / Claude Code / Codex 发现并使用的小说创作长任务 Skill，使任意 Agent 能把一部长篇或系列小说拆成有序 phase，并在压缩与换会话后继续推进而不丢失创作脉络。

## 非目标

以下内容不在本轮改造范围内：

- 直接生成某部具体小说正文或样章成品。
- 把本仓库改造成通用“任何内容创作”平台；目标聚焦小说，不泛化到论文、短视频文案或广告脚本。
- 重新设计 planctl 的底层架构或引入额外服务端能力。
- 删除长任务续跑机制本身；本次是换领域，不是拆掉流程基础设施。

## 硬性工程约束

### 平台与工具链

- 维持现有 Ruby 脚本运行前提（Ruby >= 2.6）。
- 维持 Git 工作区前提与现有 planctl 命令接口。
- 保持仓库内测试入口仍可直接运行。
- 避免为了文档改造引入额外 package manager 或构建步骤。

### 依赖边界

禁止引入以下内容：

- 新的语言运行时依赖或 gem/npm/pip 依赖。
- 把小说方法论外包给外部在线服务或闭源 API 作为前提。
- 与小说创作无关的工程框架宣传性内容。

允许使用的仅限：

- 仓库现有 Ruby / Markdown / 测试基础设施。
- 当前 references/ 与 assets/ 目录下的静态文档资源。
- 必要的仓库内脚本、说明文档与测试修改。

## 质量底线

- 顶层入口文案、模板文案、方法论文案三者要互相一致，不得出现“README 说小说，模板仍写代码迁移”的断裂。
- 关键术语要可执行：例如“悬念”“冲突”“人物弧光”不能只做概念说明，必须落成 phase/scene 级检查点。
- 示例与阶段划分要能支撑长篇项目，而不是只适用于短篇灵感记录。
- 所有最终新增文件和改动要通过至少一轮脚本/测试验证，避免只做表面文案替换。

## 其他不可跨越的边界

- Future phase 仍应使用带 PHASE_CONTRACT_PLACEHOLDER 的成对占位合同，直到轮到该 phase。
- 若 planctl 与文档语义需要同步改名或改示例，优先改用户可见语义，不随意破坏兼容命令。
- 如果发现某项建议更适合 wiki/内部页而不适合 README，应下放到 references/*，而不是把 README 写成百科全书。
