# AI 长篇小说通用工作流（Phase-Fiction Workflow）

> 一份面向长篇小说、系列小说与小说修订工程的 AI 长任务方法论。
> 它不把写作当作“灵感喷发”，而把它当作可分阶段推进、可压缩恢复、可审视质量的长期创作系统。

---

## 0. 它解决的根本问题

AI 写小说最容易在长距离任务里崩，而不是在单个段落上崩。一个模型今天也许能写出漂亮的 1000 字，但当项目跨越 5 万、10 万甚至几十万字时，会稳定出现五类失稳：

1. **故事承诺漂移**：开头答应读者的是 A，十章后小说实际上在讲 B。
2. **人物内核失真**：角色的欲望、恐惧、缺口和说话方式无法跨章节稳定保持。
3. **张力塌陷**：章节看似很多，真正推动选择、代价与悬念的场景却越来越少。
4. **设定与因果断裂**：伏笔、规则、关系网和时间线在压缩或续写后互相打架。
5. **修订层次混乱**：结构问题、人物问题、语言问题被混在同一轮里处理，导致越改越糊。

传统做法是写一份大纲、一份人物表，再让 AI “顺着写”。这种做法在单次会话里尚能勉强维持，但一旦跨小时、跨会话、跨 agent，小说最贵的部分就会丢失：故事承诺、人物弧光、悬念账本、风格秩序与修订次序。

Phase-Fiction 的核心主张是：

> **把小说长任务里需要稳定性的部分，从模型记忆中剥离出来，外部化为仓库里的创作合同。AI 不负责“凭记忆守住一切”，AI 只负责在当前小窗口里完成当前创作合同。**

## 1. 核心思想：Phase-Fiction Model

把一部长篇小说或系列创作建模为一条**有序的创作合同链**：

```
故事合同链 = [ Phase₀ → Phase₁ → Phase₂ → ... → Phaseₙ ]

每个 Phase = ⟨ 定位合同, 执行合同 ⟩
每次执行 = ⟨ 全局创作约束 + 当前阶段定位 + 当前执行边界 ⟩
      → 中间产物 / 章节 / 修订结论
      → 写回状态
```

对小说来说，这条链常见的形状不是“写第 1 章、写第 2 章、写第 3 章”，而是：

```text
题材与故事承诺
→ 角色与关系引擎
→ 世界规则 / 设定圣经
→ 主情节与冲突升级链
→ 卷/幕/篇章结构
→ 场景卡与悬念账本
→ 章节起稿波次
→ 结构修订
→ 人物修订
→ 语言与节奏精修
```

三个不变量撑起整套方法论：

| 不变量 | 含义 | 违反后果 |
|--------|------|---------|
| **I1 单一活跃合同** | 任意时刻只允许一个 phase 处于实施中 | 创作焦点漂移，同一轮里同时改结构、人物和 prose |
| **I2 三文件上下文** | 工作窗口恒定为 `common + phase + execution` 三份 | 角色设定、结构约束与当前任务被未来内容冲散 |
| **I3 完成即事实** | 只有写入外部状态的阶段才算完成 | 以为已经解决的人物线、伏笔线和修订结论无法追溯 |

这些不变量存在的目的，不是让 AI “尽量自觉”，而是让它**无法轻易失控**。

### 1.1 三层架构：通用内核、Profile 层、项目实例

为了支持不同类型的小说，同时又不把底层工作流撕裂成多套 Skill，Phase-Fiction 应严格分成三层：

1. **通用内核**：负责状态机、双层合同、恢复协议、ledger、doctor/finalize、artifact gate、仓库基础设施。

  这一层不应感知悬疑、言情、奇幻、严肃文学等题材差异。

1. **Profile 层**：负责表达不同题材 / 叙事引擎的默认 phase 图、必带产物、补充问卷、修订轮次与质量焦点。

  它回答的是“什么会变”，而不是重写调度器。

1. **项目实例层**：负责某一部小说自己的 premise、人物、世界观、beat map、章节草稿与修订结果。

  这一层是运行时生成物，不应回流成所有项目的默认值。

最重要的判断标准不是“这个东西重不重要”，而是“它是否跨题材恒定成立”。跨题材恒定成立的，放核心；会随题材和叙事引擎变化的，放 profile；只属于某一部作品的，放项目实例。

## 2. 八原则（可打印挂墙）

| # | 原则 | 一句话 | 落地动作 |
|---|------|--------|---------|
| P1 | **状态外部化** | 创作状态写到文件，不写在聊天里 | `state.yaml` 记录完成阶段与摘要 |
| P2 | **调度与执行分离** | 脚本决定下一步写什么，AI 决定如何写 | 用 `planctl` 管 phase，而不是靠会话记忆 |
| P3 | **三文件上下文律** | 全局创作约束 + 当前阶段目标 + 当前执行边界 | `manifest` 强制给出 `required_context` |
| P4 | **双层合同** | “这一阶段要解决什么”与“这一轮能碰什么”必须分开 | `phases/*` 与 `execution/*` 双文档 |
| P5 | **依赖强制校验** | 先稳故事底座，再写章节，再修 prose | `depends_on` + `--strict` 阻断跳步 |
| P6 | **完成即写入** | 未写入 ledger 的情节/修订不算完成 | `complete` 是唯一写回入口 |
| P7 | **固定恢复协议** | 压缩后永远 manifest → handoff → advance | 让续写不依赖隐式记忆 |
| P8 | **里程碑外部化** | 每个阶段都留下可追溯创作里程碑 | `complete` 自动落 state/handoff，必要时留 git 里程碑 |

## 3. 什么样的小说会抓人、耐人寻味、广受欢迎

好小说并不是“文笔很美”这么简单。能抓住读者、让人愿意继续翻页、读完后还会回味的作品，通常同时满足以下几层力量：

### 3.1 故事承诺要锐利

读者会在最初几页里问：**“这到底是个什么故事，我为什么要继续？”**

抓人的开局通常同时具备：

- **明确的 promise**：这部小说将提供哪一种情感体验、道德困境、类型快感或精神冒险。
- **不稳定状态**：主角的世界已经不稳，或即将被打破。
- **核心问题**：读者马上知道一个悬而未决的问题，而不是只有气氛没有推进力。

如果一句话说不清故事承诺，长篇几乎一定会越写越散。

### 3.2 人物必须有“欲望 + 缺口 + 代价”

受欢迎的小说很少靠设定 alone，更多靠人物驱动。人物至少要同时具备：

- **显性欲望**：他以为自己在追什么。
- **隐性缺口**：他真正需要面对什么匮乏、误解或伤口。
- **代价曲线**：每一次选择都要更贵，不能永远零损耗。

读者不是为“设定解释”追更，而是为“这个人接下来会怎么选、会付出什么”追更。

### 3.3 情节的本质不是事件堆积，而是压力升级

一个能持续吸引人的 plot engine，核心不在“大事很多”，而在：

- 每个转折都让目标更难。
- 每次揭示都改变读者对过往信息的理解。
- 每个阶段都把人物推向更不可逆的选择。

换句话说，小说真正的推进力来自：

> **未解问题 + 被迫选择 + 选择后的不可逆后果**

### 3.4 场景必须发生“变化”

场景不是“角色在某地说了几句话”，而是一个单位时间内的力量重组。一个有效场景通常至少发生一种变化：

- 信息优势改变了。
- 权力关系改变了。
- 情感温度改变了。
- 承诺与立场改变了。
- 风险与时钟被重新定义了。

如果场景结束时一切原样不动，这个场景大概率可以删。

### 3.5 悬念不是故作神秘，而是信息分配

耐人寻味的小说会精确控制读者知道什么、不知道什么、误以为什么。它依赖：

- **伏笔**：先给一个看似普通的信息点。
- **延迟解释**：暂时不揭示真正含义。
- **回报**：更晚的节点让前面的信息突然增值。
- **重解释**：揭示不是简单答题，而是反过来改变读者理解。

读者真正享受的是“原来如此，而且这样才最合理”。

### 3.6 受欢迎来自“可预期的快感 + 不可预期的走法”

商业上广受欢迎的作品，常常不是完全打破类型，而是做到两件事：

- 给足读者进入这个类型时期待的快感。
- 又在关键节点上给出新鲜、但回头看仍合理的变化。

也就是常说的：**意料之外，情理之中。**

### 3.7 耐人寻味来自“主题被选择而不是被讲解”

小说的主题不是作者在段落里解释出来，而是人物在冲突中一次次做选择之后自然显影。读者记住的往往不是“作者想表达 X”，而是：

- 哪个角色在最痛的时候仍然怎么做。
- 哪个代价最终被谁承担。
- 哪个真相来得太晚，但又非来不可。

主题应该被剧情与人物承担，而不是被独白替代。

## 4. 制品体系（Artifact Layout）

> **基础设施前提**：目标小说项目仍建议放在 git 工作区。对长篇创作来说，git 的价值不是“像工程一样管理代码”，而是让章节、设定、提纲、修订意见与阶段账本都能回溯、比较与回退。除非有明确理由，否则不要放弃这层保障。

一份完整的 Phase-Fiction 实施，最少需要以下文件：

```
<novel-project>/
├── plan/
│   ├── manifest.yaml            # 创作阶段清单：顺序 + 依赖 + required_context
│   ├── common.md                # 全局创作硬约束：题材、视角、时态、设定底线、禁区
│   ├── phases/phase-X.md        # 定位合同：这一阶段要解决什么
│   ├── execution/phase-X.md     # 执行合同：这一轮能碰什么、怎么验收
│   ├── state.yaml               # 执行账本（脚本写入）
│   ├── handoff.md               # 压缩恢复锚点（脚本写入）
│   └── workflow.md              # 流程说明
├── story/
│   ├── premise.md               # 可选：故事承诺与一句话高概念
│   ├── canon/                   # 可选：世界规则、时间线、设定圣经
│   ├── cast/                    # 可选：人物卡、关系网、欲望/缺口卡
│   ├── outline/                 # 可选：卷/幕/章节/场景矩阵
│   └── draft/                   # 可选：章节正文
└── scripts/
  └── planctl                  # 唯一调度入口
```

`story/` 只是示意，真实项目可以是 `novel/`、`chapters/`、`world/` 等。关键不在命名，而在**每个 phase 的允许改动边界必须清楚**。

## 5. 双层合同如何用于小说创作

### 5.1 定位合同回答“这一阶段是什么”

例如一个“角色与关系引擎”阶段的定位合同，不是写“把人物写得更立体”，而是写：

- 主角、对手、关键关系人各自的欲望、恐惧、误解与反向吸引力是否落盘。
- 角色之间的关系变化是否有可预期的升级轨迹。
- 哪些秘密属于作者知道、人物不知道、读者暂时不知道。

### 5.2 执行合同回答“这一轮能碰什么”

同样一个阶段，其 execution 合同应明确：

- 本轮只允许修改 `story/cast/**` 与 `story/canon/relationships.md`。
- 不允许顺手改 `story/draft/**` 的正文。
- 交付检查必须包括角色之间的冲突线、联盟线和背叛线是否都有落点。

**创作阶段最危险的失控，不是写不好，而是提前改了不该改的层。** 比如人物底座还没稳，就急着写完整章节；或者结构问题还没解，就开始句子级抛光。双层合同就是为此存在。

## 6. 调度脚本契约（planctl）

脚本语言可以不变，底层机制也可以复用。对小说工作流来说，`planctl` 的抽象接口仍然成立：

| 命令 | 作用 |
|------|------|
| `resolve <phase-id>` | 解析指定创作阶段的 `required_context` 与依赖结果 |
| `advance --strict` | 输出当前应做的内部动作：写作/修订/升级占位/进入收尾 |
| `complete <phase-id>` | 标记当前创作阶段完成，写回状态与 handoff |
| `resume --strict` | 压缩后恢复：项目概览 + handoff 快照 + 下一步 ACTION |
| `doctor` | 检查指令文件、manifest、state、handoff 是否一致 |

创作工作流与工程工作流的差异，不在于调度器，而在于**阶段定义与验收语言**。

## 7. 标准执行循环（Golden Loop）

```text
advance --strict
→ 只读取 common + 当前 phase + 当前 execution
→ 只在当前边界内完成一段创作任务
→ 用完成判定与交付检查自检
→ complete <id> --summary ... --next-focus ... --continue
→ 进入下一内部动作
```

对小说创作来说，这个循环最重要的价值有三个：

- **防止灵感发散吞掉结构**。
- **防止修订次序混乱**。
- **防止换会话后人物和设定失忆**。

## 8. 如何切小说 phase

切得好，AI 会稳定；切得差，AI 会越写越散。小说 phase 最常见的几种切法如下。

### 8.1 按故事层次切

适合从 0 开始搭一部长篇：

1. 故事承诺 / high concept
2. 角色与关系引擎
3. 世界规则 / 设定圣经
4. 主线冲突与升级链
5. 卷/幕结构
6. 场景矩阵与悬念账本
7. 章节起稿
8. 结构修订
9. 人物修订
10. 语言精修

这是最稳的切法，因为它遵循“先决定小说靠什么活，再决定小说怎么展开”。

### 8.2 按卷/幕/篇切

适合大框架已经稳住的系列作：

- 卷一结构
- 卷二结构
- 卷三结构
- 终局与回收

前提是人物引擎、故事承诺与世界规则已经基本稳定，否则会在每一卷里反复返工。

### 8.3 按起稿波次切

适合已经有完整 scene matrix 的项目：

- 第 1 波：起稿第 1–10 章
- 第 2 波：起稿第 11–20 章
- 第 3 波：起稿终局章节

每一波都必须有明确的章节范围与回收目标，不能写成“继续写正文”。

### 8.4 按修订轮次切

适合已有草稿的作品：

1. 结构修订：删线、换线、重排因果
2. 人物修订：动机、关系、弧光、声口
3. 连续性修订：时间线、设定、伏笔、回收
4. 节奏修订：场景密度、章节断点、悬念分配
5. 语言修订：句子、意象、节拍、重复词

**结构问题永远先于文风抛光。** 这是小说修订最常见也最昂贵的次序错误。

### 8.5 切 phase 的三个判据

一个合格的小说 phase 应该同时满足：

1. **可独立交付**：产物能落盘，例如人物卡集、场景矩阵、某一波章节草稿、某一轮修订 memo。
2. **可客观检查**：能写出勾选项，而不是“写得更有感觉”。
3. **尺寸适中**：单次 AI 实施窗口大致 30–120 分钟内可完成。

写不出客观完成判定，就说明切错了。

## 9. 如何把“抓人”和“耐人寻味”写成检查项

下列抽象审美词，必须翻译为操作性检查。

### 9.1 “抓人”不是形容词，是推进力检查

可操作检查示例：

- 每一章都显式延续或加压一个未解问题。
- 每个场景都迫使至少一个角色作出更贵的选择。
- 每个章节末尾都存在新的不稳定性，而不是简单总结。

### 9.2 “耐人寻味”不是晦涩，而是可回看增值

可操作检查示例：

- 关键揭示会反向增值前文至少一个细节。
- 主题通过人物选择显影，而不是靠作者解释。
- 重要 motif 至少出现两次不同语境，并在后段获得回收。

### 9.3 “广受欢迎”不是迎合所有人，而是兑现类型承诺

可操作检查示例：

- 目标读者为什么会打开这部小说，前三章是否已经兑现一部分承诺。
- 作品的主要快感来源是什么：悬疑、情感、权谋、成长、奇观、黑色幽默，是否稳定供给。
- 反转是否建立在前文已经给出的合法信息上，而不是强行扭转。

## 10. `common.md` 在小说项目里该约束什么

小说项目的 `common.md` 尤其适合放以下长期稳定约束：

- 目标题材 / 类型混合比例
- 叙事视角与时态纪律
- 主角不可被破坏的核心设定
- 世界规则与超自然/科技规则底线
- 暴力、伦理、尺度与内容禁区
- 语言风格底线（如克制/密实/冷幽默/高张力）
- 目标篇幅、卷数、章节粒度上限
- 不做事项（例如“不写群像支线扩张到抢走主线”）

如果这些约束是某类小说的默认要求，而不是每个长篇都必然成立，就不应直接写死在通用模板里，而应由 profile 层注入。

不该放进 `common.md` 的，是只对某一阶段成立的局部策略，例如“先写卷一再写卷二”。那属于 phase 合同，不属于永恒约束。

## 11. 仓库级 AI 指令为什么重要

小说项目看似“更自由”，其实更依赖强制层。没有 `.github/copilot-instructions.md` / `CLAUDE.md` / `AGENTS.md` 这三份同步指令，AI 会在几轮会话后重新退化为：

- 自行决定当前应该写哪一章。
- 一次性打开所有大纲、人物、章节文件。
- 在没有完成上一轮修订的情况下直接写下一轮正文。
- 用感觉宣布“人物已经立住了”。

所以对小说工作流来说，强制层不是束缚创意，而是保护创意不被混乱吞掉。

## 12. 评估口径（Evaluation Vocabulary）

| 术语 | 含义 |
| --- | --- |
| **故事承诺 / story promise** | 读者一开始被承诺的核心快感、问题或精神冒险 |
| **可持续张力** | 不靠单次爆点，而靠持续升级的选择与代价推动阅读 |
| **场景有效** | 场景结束后，信息、关系、风险、立场中至少一项发生变化 |
| **设定连续性** | 角色、关系、世界规则、时间线、伏笔前后一致 |
| **修订闭环** | 每一轮修订都只解决一种主问题，并有明确的完成判定 |
| **无损续跑** | 换会话后通过 manifest → handoff → advance 能恢复到正确创作阶段 |

---

## 13. 最后的判断标准

Phase-Fiction 不承诺“AI 一定写出伟大小说”。它承诺的是更基础、也更关键的东西：

> **当项目足够长时，小说最宝贵的秩序不会在压缩、换会话和多轮修订中悄悄蒸发。**

一部小说最终是否动人，仍取决于作者/团队对人物、选择、主题与节奏的判断；但如果没有这套外部化的执行秩序，AI 大概率连“稳定地把故事写完并守住它本来该是什么”都做不到。
