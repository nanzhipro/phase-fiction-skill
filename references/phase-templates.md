# Fiction Phase 双层合同模板

每个 fiction phase 需要两份文档：**定位合同** `plan/phases/phase-X.md` 和 **执行合同** `plan/execution/phase-X.md`。职责不可合并。

---

## 1. 定位合同：`plan/phases/phase-X-<slug>.md`

回答“这个阶段**是什么**”。

```markdown
# Phase X: <标题>

## 阶段定位

<这个阶段在整条创作链路里解决什么问题，一句话>

## 必带上下文

- plan/common.md
- Phase X-1 已完成

## 阶段目标

- <可验证的目标 1>
- <可验证的目标 2>
- <可验证的目标 3>

## 实施范围

- <允许涉及的故事层/文件/目录 1>
- <允许涉及的故事层/文件/目录 2>

## 本阶段产出

- <具体交付物 1，例如故事承诺文档>
- <具体交付物 2，例如角色关系表>

## 明确不做

- <最容易误伤的越界项 1>
- <最容易误伤的越界项 2>

## 完成判定

- <客观检查 1>
- <客观检查 2>
- <客观检查 3>

## 依赖关系

- 依赖 Phase X-1
```

### 小说阶段的完成判定应该怎样写

不要写：

- “人物更立体了”
- “开头更抓人了”
- “章节节奏更好”

要写成：

- 主角、对手、关键关系人各自的欲望、恐惧、缺口与误信念都已落盘。
- 每个场景卡都包含 POV、目标、阻力、变化与章节断点。
- 第一卷每章末尾都保留一个新的不稳定性或未解问题。

### 定位合同写作铁律

- **阶段目标必须可验证**。写不出验证方式的目标，是无效目标。
- **“明确不做”通常比“阶段目标”更重要**，它保护边界，优先写。
- **完成判定禁止主观词**：不要“更精彩”“更高级”“更有感觉”。

## 2. 执行合同：`plan/execution/phase-X-<slug>.md`

回答“这次实施**能碰什么、不能碰什么**”。

```markdown
# Phase X 执行包

本文件不能单独使用。执行 Phase X 时，必须同时携带完整的 `plan/common.md` 和 `plan/phases/phase-X-<slug>.md`。

## 必带上下文

- `plan/common.md`
- `plan/phases/phase-X-<slug>.md`

## 执行目标

- <本次实施要解决的具体问题 1>
- <本次实施要解决的具体问题 2>

## 本次允许改动

- <路径/目录 1，例如 `story/cast/**`>
- <路径/目录 2，例如 `story/canon/relationships.md`>
- <路径/目录 3>

## 本次不要做

- <禁止涉及的层次，例如 draft 正文>
- <禁止顺手改动的阶段外内容>
- <禁止引入的偷懒方式，例如直接重写主线承诺>

## 交付检查

- <客观可勾选项 1>
- <客观可勾选项 2>
- <客观可勾选项 3>

## 执行裁决规则

- 如果 <某触发条件>，直接判定无效并回退
- 如果引入超出本 phase 范围的范围漂移，直接回退到规划边界
- 如果把结构问题和 prose 问题混在同一轮处理，直接判定无效
```

### 小说执行合同写作铁律

- **execution 是围栏，不是写作课大纲**。不要写“先构思、再发挥、再润色”这种泛泛步骤。
- **“允许改动”必须是路径级白名单**，例如 `story/outline/**`，而不是“和大纲相关的内容”。
- **交付检查比完成判定更细**，用于判断这一次创作/修订是否真的完成。
- **修订阶段必须单一主问题**：结构轮只修结构，人物轮只修人物，语言轮只修语言。

## 3. 示例：人物引擎阶段

### 3.1 定位合同示例

```markdown
# Phase 1: Build the cast engine

## 阶段定位

为主角群、对手与关键关系人建立稳定的人物驱动系统，作为后续情节与场景的动力源。

## 阶段目标

- 为核心角色建立欲望、恐惧、缺口、误信念与代价曲线。
- 明确主角与对手的冲突来源、关系张力与相互吸引/排斥机制。

## 本阶段产出

- 角色卡
- 关系网图
- 人物冲突升级表

## 完成判定

- 主角、对手、关键关系人都具备显性欲望与隐性缺口。
- 每对关键关系都写明当前状态、潜在裂缝与升级方向。
- 没有任何角色只以“功能人”或“设定解释器”存在。
```

### 3.2 执行合同示例

```markdown
# Phase 1 执行包

## 本次允许改动

- `story/cast/**`
- `story/canon/relationships.md`

## 本次不要做

- 不写 `story/draft/**` 正文
- 不修改卷结构和章节顺序

## 交付检查

- 每个核心角色文档都包含 want / need / fear / misbelief / cost。
- 每条关键关系都包含当前关系、隐藏张力、未来爆点。
- 对手不是单纯反派标签，而有自己的目标与逻辑。
```

## 4. 未来 phase 的占位合同（未轮到该 phase 时使用）

当 manifest 需要提前引用未来 phase 的 `plan_file` / `execution_file`，但当前还没进入该 phase 时，先为这一对文件写**占位合同**。作用有两点：

- 让 `planctl doctor` 能验证 manifest 引用的文件存在。
- 让 `planctl advance --strict` 在该 phase 变成当前 phase 时返回 `ACTION: promote_placeholder`，逼 agent 先补正式合同，但不把 phase 边界变成用户确认点。

### 4.1 占位定位合同

```markdown
# Phase X: <标题>

> PHASE_CONTRACT_PLACEHOLDER
> 当前为占位合同，禁止实施。
> 进入本 phase 时，必须先把本文件和配对 execution 文件升级成正式合同，再开始任何实现。

## 当前状态

- 当前尚未进入本 phase
- 本文件仅用于保留 phase 标题与 manifest 引用
```

### 4.2 占位执行合同

```markdown
# Phase X 执行包

> PHASE_CONTRACT_PLACEHOLDER
> 当前为占位合同，禁止实施。
> 进入本 phase 时，必须先把本文件和配对 phases 文件升级成正式合同，再开始任何实现。

## 当前状态

- 当前尚未进入本 phase
- 本文件不是执行边界，不能直接用于实施
```

### 4.3 占位合同铁律

- `PHASE_CONTRACT_PLACEHOLDER` 必须保留在文件前 40 行，直到正式合同写完再删除。
- phase 真轮到当前时，先同时升级 `phases/*` 和 `execution/*`，再 rerun `advance --strict`。
- 不要只升级一份；单边升级会让边界和目标失步。

## 5. 自检清单

- [ ] 定位合同里的阶段目标每一条都有验证方式。
- [ ] “明确不做”列出了最容易误伤的层次或文件。
- [ ] 完成判定里没有主观词。
- [ ] 执行合同的允许改动是路径级白名单。
- [ ] 本次不要做排除了其他 phase 的范围。
- [ ] 交付检查每一条都能勾或不勾，没有灰色地带。
- [ ] 执行裁决规则里至少包含一条“范围漂移立即回退”。
- [ ] 两份文档的必带上下文都包含 `plan/common.md` 和对方。
- [ ] 整体工作量估计在 30–120 分钟的 AI 实施窗口内。

任一项不过，重写。

---

## 6. 常见反模式

| 反模式 | 表现 | 修正 |
|--------|------|------|
| 完成判定写“人物更鲜活了” | 无法证明 | 改为欲望/缺口/代价是否落盘 |
| 允许改动写“和剧情相关的文件” | 不是白名单，AI 会发散 | 改为具体路径 |
| 在人物阶段顺手写正文 | 底座未稳先起稿 | 退回边界，只改 cast/canon |
| 把结构修订和句子润色混在一起 | 越改越糊 | 分成不同修订 phase |
| phase 文档过短 | 验收点不够 | 补足目标、边界与完成判定 |
| common.md 写“第一章先怎么写” | 把策略写成长期约束 | 下放到对应 phase |
